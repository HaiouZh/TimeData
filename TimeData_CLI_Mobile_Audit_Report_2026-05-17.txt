# TimeData CLI 和 Mobile 包审计报告

审计日期: 2026-05-17
审计范围: packages/cli 和 packages/mobile

## 一、CLI 包 (packages/cli) 审计

### 1.1 命令结构与参数解析

问题 1: 参数解析缺乏严格验证
文件: packages/cli/src/lib/args.ts (第 1-23 行)
问题描述:
- parseFlags() 函数对参数值没有进行类型验证或长度限制
- 允许任意长度的标志值，可能导致内存溢出或注入攻击
- 不处理重复标志的情况（后面的值会覆盖前面的值，无警告）

建议改进方向:
- 添加最大值长度限制（如 10KB）
- 对特殊字符进行转义或验证
- 记录重复标志的警告日志
- 实现白名单验证已知标志名称

优先级: 中

---

### 1.2 配置与凭据处理

问题 2: 敏感信息可能暴露在日志中
文件: packages/cli/src/lib/config.ts (第 18-49 行)
问题描述:
- resolveConfig() 返回的 token 可能被打印到日志或错误消息中
- 没有对敏感信息的脱敏处理
- 配置文件路径在错误消息中完整显示

建议改进方向:
- 实现敏感信息脱敏函数（如 token 只显示前后各 4 字符）
- 在错误消息中使用通用路径而非完整路径
- 添加日志级别控制，生产环境禁用详细日志

优先级: 高

---

问题 3: 配置文件权限未验证
文件: packages/cli/src/lib/config.ts (第 25-32 行)
问题描述:
- readFileConfig() 读取配置文件时不检查文件权限
- 在多用户系统上，其他用户可能读取包含 token 的配置文件
- 没有验证配置文件所有者

建议改进方向:
- 检查配置文件权限（应为 0600）
- 验证文件所有者是当前用户
- 如果权限不安全，发出警告或拒绝读取
- 在 Windows 上使用 ACL 检查

优先级: 高

---

### 1.3 API 交互与错误处理

问题 4: HTTP 错误处理不完整
文件: packages/cli/src/lib/api-client.ts (第 12-37 行)
问题描述:
- 只特殊处理 401 错误，其他 4xx/5xx 错误处理不足
- 没有重试机制，网络抖动会导致失败
- 响应超时没有设置，可能无限等待
- 没有处理 JSON 解析失败的情况（第 25 行 .catch(() => null) 过于宽泛）

建议改进方向:
- 为不同的 HTTP 状态码添加特定处理（403、429、5xx）
- 实现指数退避重试机制（最多 3 次）
- 设置请求超时（建议 30 秒）
- 区分 JSON 解析错误和网络错误
- 添加详细的错误日志

优先级: 高

---

问题 5: Bearer Token 格式未验证
文件: packages/cli/src/lib/api-client.ts (第 15-16 行)
问题描述:
- 直接将 token 拼接到 Authorization 头，没有验证格式
- 空 token 会导致发送 "Authorization: Bearer " 头
- 没有检查 token 是否包含非法字符

建议改进方向:
- 验证 token 格式（如长度、字符集）
- 如果 token 为空，不发送 Authorization 头
- 对 token 进行基本的格式检查

优先级: 中

---

问题 6: URL 构造存在注入风险
文件: packages/cli/src/lib/api-client.ts (第 14 行)
问题描述:
- URL 构造使用字符串拼接，虽然有基本处理但不够完整
- list.ts 中使用 encodeURIComponent() 但其他地方没有

建议改进方向:
- 统一使用 URL 类进行 URL 构造
- 对所有查询参数进行编码
- 验证 serverUrl 格式

优先级: 中

---

### 1.4 命令实现

问题 7: log 命令缺少必要的验证
文件: packages/cli/src/commands/log.ts (第 4-32 行)
问题描述:
- 没有验证 category 格式（可能包含特殊字符）
- note 字段没有长度限制

建议改进方向:
- 验证 category 路径格式

- 添加用户确认提示（可选）

优先级: 中

问题 8: doctor 命令的检查不完整
文件: packages/cli/src/commands/doctor.ts (第 38-90 行)

- 只检查读权限，没有检查写权限
- 没有检查 API 版本兼容性
- 没有检查网络延迟或性能问题

建议改进方向:
- 添加写权限检查（可选的 POST 请求）
- 检查 API 版本信息
- 测量响应时间并警告慢速连接
- 检查 SSL 证书有效性

优先级: 中

## 二、Mobile 包 (packages/mobile) 审计

### 2.1 Capacitor 配置

问题 10: 安全配置不够严格
文件: packages/mobile/capacitor.config.ts (第 1-17 行)
问题描述:
- allowMixedContent: false 正确，但没有其他安全配置
- 没有配置 CSP（Content Security Policy）

建议改进方向:
- 添加 CSP 配置
- 在生产环境禁用调试
- 配置允许的域名白名单
- 添加 CORS 配置

优先级: 高

---

### 2.2 Android 配置

问题 11: 严重的安全问题 - cleartext traffic 启用
文件: packages/mobile/android/app/src/main/AndroidManifest.xml (第 5 行)
问题描述:
- android:usesCleartextTraffic="true" 允许明文 HTTP 通信
- 这在生产环境中是严重的安全风险
- 可能导致中间人攻击和数据泄露

建议改进方向:
- 在生产环境设置为 false
- 使用网络安全配置文件（network_security_config.xml）
- 仅在开发环境允许 cleartext
- 强制使用 HTTPS

优先级: 严重

---

问题 12: 备份功能启用
文件: packages/mobile/android/app/src/main/AndroidManifest.xml (第 4 行)
问题描述:
- android:allowBackup="true" 允许应用数据备份
- 可能导致敏感数据（token、数据库）被备份到云端
- 恢复时可能恢复过期的凭据

建议改进方向:
- 设置为 false
- 或使用 android:backupAgent 指定自定义备份代理
- 排除敏感数据文件
- 实现备份加密

优先级: 高

---

问题 13: 文件访问权限过于宽泛
文件: packages/mobile/android/app/src/main/res/xml/file_paths.xml (第 1-5 行)
问题描述:
- <external-path name="my_images" path="." /> 允许访问整个外部存储
- <cache-path name="my_cache_images" path="." /> 允许访问整个缓存目录
- 可能导致敏感文件被其他应用访问

建议改进方向:
- 限制路径到特定目录（如 path="images" 而非 path="."）
- 使用最小权限原则
- 定期审查文件访问权限

优先级: 高

---

问题 14: 最小 SDK 版本过低
文件: packages/mobile/android/variables.gradle (第 1 行)
问题描述:
- minSdkVersion = 24 (Android 7.0) 已过时
- 缺少现代 Android 安全特性
- 支持的设备中许多已停止安全更新

建议改进方向:
- 升级到 minSdkVersion 26 或更高（Android 8.0+）
- 考虑升级到 28+（Android 9.0+）
- 评估用户设备分布后决定

优先级: 中

---

问题 15: ProGuard 配置不完整
文件: packages/mobile/android/app/proguard-rules.pro (第 1-23 行)
问题描述:
- ProGuard 规则几乎为空，只有注释
- 没有保护关键类和方法
- 没有配置混淆规则

建议改进方向:
- 添加 Capacitor 相关的保护规则
- 保护 API 接口类
- 配置混淆规则以减小 APK 大小
- 保留调试信息用于崩溃分析

优先级: 中

---

### 2.3 构建脚本

问题 16: 签名密钥管理不安全
文件: packages/mobile/android/app/build.gradle (第 2-6 行)
问题描述:
- 签名密钥通过 Gradle 属性传递，可能暴露在构建日志中
- 没有验证密钥文件是否存在
- 没有检查密钥有效期

建议改进方向:
- 使用环境变量而非 Gradle 属性
- 实现密钥文件验证
- 添加密钥有效期检查
- 使用密钥管理服务（如 Google Play App Signing）

优先级: 高

---

问题 17: 版本号管理不规范
文件: packages/mobile/android/app/build.gradle (第 7 行)
问题描述:
- versionCode 使用时间戳格式 (260513)，容易冲突
- versionName 直接使用 versionCode，不符合语义版本
- 没有自动化版本管理

建议改进方向:
- 使用递增的整数作为 versionCode
- 使用语义版本作为 versionName（如 0.1.0）
- 在 CI/CD 中自动管理版本号
- 记录版本变更日志

优先级: 中

---

问题 18: 构建脚本缺少错误处理

## 八、风险评估

### 高风险项
1. cleartext traffic 启用 - 可能导致数据泄露
2. 备份功能启用 - 可能导致凭据泄露
3. 混淆禁用 - 可能导致代码被反编译
4. 签名密钥暴露 - 可能导致应用被篡改
5. 敏感信息暴露 - 可能导致凭据泄露

### 中风险项
1. 参数验证不足 - 可能导致注入攻击
2. 错误处理不完整 - 可能导致信息泄露
3. 文件权限过宽 - 可能导致数据被访问
4. 最小 SDK 版本低 - 缺少安全特性

### 低风险项
1. 测试覆盖不足 - 可能导致未发现的 bug
2. 文档不完整 - 可能导致误用
3. 版本管理不规范 - 可能导致混淆
