# ADR 0002 — Sync 与 Backup 是两件事

## 状态

Accepted。Web Backup 完整版已上线（Phase 5.x），Android 可见目录备份在 Phase 5.4 计划中。

## 背景

TimeData 同时需要"多设备同步"和"数据备份"两种能力。在功能规划时，自然的诱惑是把它们合并：

> "我们已经有 server 了，每次同步就把数据上传一份，那不就同时是备份吗？"
> "用户恢复 backup 时直接覆盖服务器，多设备会拉到新版本，是不是顺便就同步了？"

这种合并看起来省力，但语义会出问题。

## 决策

**Sync 和 Backup 是独立能力，互不蕴含、互不替代。**

- **Sync**：多设备之间同步当前数据。要解决的问题是"同一个用户的两台设备数据不一致"。
- **Backup**：防丢失的快照。要解决的问题是"用户误删 / 浏览器清缓存 / 服务器跑路 / 迁移失败"。

### 具体规则

1. **同步不算备份**。即使数据已经同步到服务器，仍然推荐用户定期手动导出 Backup JSON。理由见下。
2. **恢复 Backup 不会自动覆盖服务器**。`importBackup` 清空本地 + 写入备份数据 + 清掉 `LAST_SYNCED_KEY` + **同时清空本地 syncLog**。下次同步时本地没有任何"待 push"日志——服务器现状会被拉到本地。这是**故意**的行为：让用户先看清楚两边都是什么，再人工决策。
3. **服务端在每次 push 前自动产一份备份 `.db`**。这是 sync 行为内嵌的安全网，不是用户的"主备份"。

## 理由

### 为什么同步不算备份

- **服务器可能丢数据**：硬盘坏、容器误删、自更新失败。Web 端的 Backup JSON 让用户在自己机器上有副本。
- **同步不保留历史**：服务器只有最新状态。误删一条记录、几分钟后所有设备都没了。Backup JSON 是某个时间点的快照。
- **格式可移植**：JSON 文件，将来换技术栈、迁移到新工具都能读。SQLite 数据库相对绑定。
- **跨账号 / 跨实例**：用户换台服务器、重装、迁移，靠的是 Backup JSON 而不是同步。

### 为什么恢复 Backup 不自动覆盖服务器

如果"恢复就 push"：

1. **多设备灾难**：用户恢复了一份 3 周前的备份后立即同步，3 周内的最新数据全没了，且其他设备拉过去也是旧数据。
2. **"恢复"语义被污染**：用户预期"恢复 = 让本机回到那个时间点"，实际却变成"把所有人的设备拉回那个时间点"。
3. **不可逆**：备份恢复 + 自动同步 + 多设备拉取一气呵成，用户来不及反应。

显式的两步（恢复本地 → 用户决定是否同步）让用户保留控制权。代价是用户要多操作一次，但这一次操作是"有意识的"。

## 实现要点

- `importBackup`（`packages/client/src/backup/importBackup.ts`）：清空 + 写入 + 清 syncLog + 清 lastSync key。**不调任何同步函数**。
- UI 层（`packages/client/src/pages/settings/SettingsDataPage.tsx`）应明确提示用户："恢复完成，服务器仍是恢复前的数据。如需同步，请手动操作并谨慎选择同步方向。"
- 服务端 `createServerBackup`（`packages/server/src/sync/backup.ts`）：每次 sync push 前自动产一份 `.db`。这是给运维的兜底网，不替代用户的 Backup JSON。

## 替代方案为什么被否

**"恢复时弹窗问用户是否同步"**：弹窗 UI 容易引导用户随手点确认，依然有"覆盖服务器"的风险。不如直接默认不同步，让用户主动去同步页操作。

**"两套合并：每次同步自动写一份本地 backup"**：实际不解决问题（服务器丢了仍然没辙），还增加用户存储负担。客户端有 `autoBackups` 表（最近 7 份）属于这个思路的轻量版本——只在浏览器里、不替代手动导出。

## 后果

**正面**：

- 语义清晰，用户预期可控。
- 极端情况（服务器跑路、IndexedDB 损坏）有明确的恢复路径。
- 跨工具/跨实例迁移有明确机制。

**负面 / 成本**：

- 用户需要"理解"这两个概念的区别（UI 文案要持续打磨）。
- 恢复后需要手动同步，多了一步。
- 维护两套机制（同步引擎 + 备份格式）。
