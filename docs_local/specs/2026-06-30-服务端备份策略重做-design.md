# 服务端备份策略重做 — 设计

- 日期：2026-06-30
- 状态：设计待评审
- 影响面：`packages/server/src/sync/backup.ts`、`packages/server/src/routes/sync.ts`、`packages/server/src/routes/admin/*`、`packages/client` 设置页（数据页 + admin 面板）、`docs/evergreen/backup.md`
- 关联：[[backup.md]]、ADR 0002（Sync ≠ Backup）

## 1. 背景与问题

生产机 `/opt/timedata/data/backups` 已膨胀到 **4.6GB / 1034 个文件**，而主库只有 6.4MB。诊断（已核对代码）：

- **每次 `/api/sync/push` 校验通过后、写入前都整盘热拷一份**（[sync.ts:472](packages/server/src/routes/sync.ts)，普通路径 `operation = sync_push`，现在每份约 6.4MB）。
- `classifyBackupRetention()`（[backup.ts:99-108](packages/server/src/sync/backup.ts)）对 15 天内非 protected 一律返回 `recent`；cleanup 保留**全部** `recent`。
- 前台 SSE 实时同步让 push 极其频繁（单日见过 138 个），15 天窗口自然攒成几个 GB。

**根因不是 cleanup 没跑，而是策略本身**：①「每次 push 都全量拷」在源头制造海量近重复快照；②「最近 15 天全留」在保留端不收敛。

### 1.1 一个关键认识

`applyChange` 跑在 `db.transaction()` 里（[sync.ts:474](packages/server/src/routes/sync.ts)），**写失败自动回滚、库不损坏**。所以"每次 push 前全量备份"对崩溃恢复价值很低；它真正的价值是**逻辑恢复**（"某次同步错误覆盖/删了数据，想回到那之前"），而逻辑恢复**不需要每次 push 的颗粒度**。真正有风险的操作（seq 冲突、force-push、时间记录重叠删除）才需要"破坏前现场"。

## 2. 目标与非目标

**目标**
- 把备份**创建**从"每次 push"压到"仅危险操作 + 用户手动 + 每日定时一次"，磁盘不再随 push 增长，热路径不再有 6.4MB I/O churn。
- 把**保留**从"最近 15 天全留 + 15 天窗口尾巴"改成"扁平 N 天窗口（默认 7） + protected 永久 + admin 手删"。
- 部署后自动把现有 4.6GB 存量裁到几十 MB。
- 顺带修复 force-push 备份**漏标 protected** 的现有 bug。

**非目标**
- 不做异地/离线备份（这些 `.db` 与主库同机，防不了磁盘损坏；离线另议）。
- 不改备份文件格式、不改恢复语义、不动 ADR 0002（恢复不自动覆盖服务器）。
- 不引入备份加密 / 机密域（见 `docs_local/ideas/2026-06-14-备份加密与机密域路线.md`，独立轨道）。
- 不把"只捕捉被删行"的轻量增量备份做进来（列为未来增强，见 §9）。

## 3. 创建策略（什么时候拍备份）

普通 push 默认**不拍**。只有以下场景拍备份；其中"危险操作"一律在写入前同步拍、并标 `protected`：

| 场景 | operation | protected | 时机 |
|---|---|---|---|
| 普通快进 push / merge 不重叠 | —（不拍） | — | — |
| seq 冲突：`local_wins_non_fast_forward` | `sync_local_wins` | 是 | apply 前（不变） |
| seq 冲突：`unknown_base`（baseSeq 为 null） | `sync_unknown_base` | 是 | apply 前（不变） |
| **时间记录重叠删除**（push 会删掉重叠时间段记录） | `sync_overlap_delete` | 是 | **apply 前探测到 →拍（新）** |
| force-push 整盘覆盖 | `sync_force_push` | **是（修 bug）** | apply 前 |
| 用户点「立即备份」 | `manual` | 是 | 立即 |
| 每日定时（有改动的日子） | `auto_daily` | 否 | 定时任务，见 §5 |

### 3.1 重叠删除前移探测（核心改动）

现状：`time_entries` 的 apply 钩子里 `deleteOverlappingEntries()`（[domains.ts:312](packages/server/src/sync/domains.ts)）会删掉与新记录时间段重叠的**其它**服务器记录，把被删 id 放进 `overriddenRecordIds`；push 路由**事后**据此把（总是已创建的）备份标 protected（[sync.ts:496-515](packages/server/src/routes/sync.ts)）。

普通 push 不再拍后，这条"事后标记"就没有备份可标、现场也已被删。因此：

- 把"是否会删重叠记录"的判断**前移到 apply 之前**：对 push 内每条 `time_entries` upsert，用与 `deleteOverlappingEntries` 同一套重叠条件先做 `SELECT`（只读探测，不删）。
- 任一条会删 → 在 apply 前创建 `sync_overlap_delete` 备份并标 protected，details 记下将被删的 record id。
- 探测逻辑与真实删除逻辑**必须同源**（抽出一个 `findOverlappingEntryIds(db, data)` 纯查询，被探测与删除复用），避免判断与实际行为漂移。

> 正常顺序记时不产生重叠，探测命中极少，不会重新引入膨胀。

### 3.2 push 路由改写要点

- 先 `analyzePushBaseSeq`（已在 apply 前，[conflict.ts](packages/server/src/sync/conflict.ts)）判 seq 冲突；再跑重叠探测。
- 需要拍备份的场景（seq 冲突 / 重叠删除）才 `createServerBackup(...)` 并 `markServerBackupProtected(...)`；普通 push **不调** `createServerBackup`。
- `SyncPushResponse.backupId` 在无备份时为 `null`（类型已允许）。`outcomeFromApplyResult` 的 `backupId` 同理在无备份时为 `undefined`。
- 删除事后 `protectedBackupDetails`/`protectedOutcomeIds` 那套"按 outcomes 标保护"的逻辑（被 §3.1 前移探测取代）；保留 outcomes 里 `overriddenRecordIds` 作为返回给客户端的信息。
- force-push 路由（[sync.ts:395](packages/server/src/routes/sync.ts)）在 `createServerBackup("sync_force_push")` 后补 `markServerBackupProtected(backup.id, { protected: true, reason: "force_push_overwrite", details })`。

## 4. 保留策略（cleanup 怎么裁）

把分层 GFS 退化成**扁平 N 天窗口**：

```
classifyBackupRetention(createdAt, protected, now, retentionDays):
  protected           → "protected"   // 永久
  age ≤ retentionDays → "recent"      // 留
  否则                 → "deletable"   // 删
```

- `cleanupServerBackups(now, retentionDays)`：保留所有 protected 和所有 `recent`，删其余；**不再有** 15 天窗口分桶 / 周尾巴 / 年尾巴。
- `retentionDays` 默认 **7**，由 §6 配置驱动。
- **孤儿清理（按文件名时间一并删 + protected 类保护闸）**：cleanup 同时扫描备份目录。磁盘上存在、manifest 里没有的 `.db`，先从文件名前缀解析 `operation`：
  - **已知 protected 类**（`sync_force_push` / `sync_local_wins` / `sync_unknown_base` / `sync_overlap_delete` / `manual*`）即便成了孤儿也**不删**（可能是丢了 manifest 登记的受保护备份，误删=数据丢失），只记日志提示人工核对。
  - 其余非保护类孤儿（`sync_push` / `auto_daily`，存量 bulk 正是 `sync_push`）按文件名解析的时间套 N 天窗口删。
  - manifest 里有、磁盘文件已不存在的条目顺手清掉。
  - 无法解析 operation 的孤儿，保守起见**不删**只记日志。
- 触发点不变（`createServerBackup` 内部 + 服务启动时）；普通 push 不再拍后，cleanup 主要靠"危险/手动备份 + 启动 + 定时日备"触发，频率足够。

**稳态估算**：非 protected 基本只有 `auto_daily`（≤1/有改动日），7 天 ≈ 7 份 ≈ 45MB；加 protected（手动 + 危险，量本就少）。

## 5. 每日定时备份（替代"首次 push 日备"）

服务端"每日滚动还原点"，不进 push 热路径。**解耦设计**：把"该备就备"抽成幂等函数 `runDailyBackupIfDue(now)`，由两条腿调用——进程内 timer + admin 手动触发端点（见 §7）。万一进程内 timer 不可靠，外部 host cron 打那个端点就是 drop-in 替代，不返工。

- **幂等函数 `runDailyBackupIfDue(now)`**：可重复调，只在"该备且未备"时真拍；多次调用/并发调用至多产出当天一份。
- **进程内 timer（挂钟判断）**：不用"每 24h 跑一次"（避免容器重启重置计时器）。`setInterval` 每数分钟调一次 `runDailyBackupIfDue()`——函数内部判断 `本地时间 ≥ 配置时点` 且 `今天还没有 auto_daily`。
- **跳过未改动**：函数内比较 `getLatestSeq()` 与"上次日备 seq"（**存 manifest.json 顶层 meta**，见 §6）。无变化则跳过——一周里不塞重复快照，出差不开 app 的日子不空拍。
- **启动补判**：服务启动时跑一次同样判断，补上停机错过的当天日备。
- `auto_daily` **不 protected**，受 §4 的 N 天窗口约束。
- 时区用 app-local（`packages/server/src/lib/timezone.ts`），"04:00" 指用户本地时间。

> 危险操作 / 重叠删除 / 手动按钮 仍在写入前**同步**拍（抓"破坏前现场"），不归定时任务管。两者职责分离。

## 6. 配置（admin 面板，master token）

服务器全局运维配置，**存服务端 `manifest.json` 顶层 `meta`，服务器自己拥有**，不进同步 `settings` 域、不进 DB 快照、不被任意同步设备改、启动即可读（去风险结论，见 §13）。

manifest.json 形状从 `{ backups }` 扩成：

```jsonc
{
  "backups": { /* ...备份条目，不变 */ },
  "meta": {
    "dailyBackup": { "enabled": true, "timeOfDay": "04:00" },
    "retentionDays": 7,
    "lastDailySeq": 0          // 上次日备时的 getLatestSeq()，供"跳过未改动"判断
  }
}
```

- `readBackupManifest()` / `writeBackupManifest()` 扩成保留并读写 `meta`；旧 manifest 无 `meta` 时回退缺省。
- 新 `GET /api/admin/backup-config` / `PUT /api/admin/backup-config`（master token）读写 `meta` 的配置部分（不暴露 `lastDailySeq` 这类内部状态）。
- admin 面板加一小块"备份设置"：开关、时点、保留天数。
- 缺省值：`enabled=true, timeOfDay="04:00", retentionDays=7`；非法值回退缺省。

## 8. 存量 4.6GB 清理

无需手写一次性脚本：**部署新版后服务启动即跑一次新 cleanup**，1012 个旧 `sync_push`（非 protected）按 N 天窗口裁——7 天前的全删，7 天内的留下（但它们都是 6.4MB 近重复，量少了之后无碍）；孤儿文件按 §4 的 protected 类保护闸一并收。预计从 4.6GB 降到几十 MB 级。

> **去风险结论：直接真删，无 dry-run 闸**（用户拍板，§13）。因此没有"先报告再删"的兜底，正确性全压在单测上——`cleanupServerBackups` 的时区窗口边界、protected 识别、孤儿文件名解析与 protected 类保护闸**必须有充分单测**（见 §10），合并前跑无参全量 server 套件。

> 验证：部署后查 `du -sh /opt/timedata/data/backups` 与文件数；查日志 `[backup] cleanup removed old backups` 的 `removedCount`。

## 9. 风险与权衡

- **只留 7 天自动备份**：远期回溯（>7 天）只能靠 protected 的手动/危险备份。已通过"手动按钮永久保留 + 大改前手动拍一份"的习惯覆盖；N 可调大。
- **重叠删除前移探测**：给 push 路径加一次只读 `SELECT`（命中重叠才拍）。开销小且只在有 `time_entries` upsert 时；探测与删除同源以防漂移。
- **进程内定时任务**：容器重启用挂钟判断 + 启动补判兜底；不依赖外部 cron，少一处运维。
- **同机备份防不了磁盘损坏**：本设计不解决，离线/异地备份是独立议题。

## 12. 落地顺序（单份 plan，内部按此序）

去风险结论：**不分阶段，一份 plan 写全**（§13）。但 plan 内部仍按下序推进，**保留侧先行**，让"止血能力"最早就位、可单独验证：

1. 保留侧重做（`classifyBackupRetention` 扁平化 + cleanup 孤儿对账与 protected 类保护闸 + manifest `meta` 扩展 + 测试）——**先把存量止血能力就位**。
2. 创建侧改写（普通 push 不拍 + 重叠删除前移探测 + force-push 修保护 + 测试）。
3. 每日定时任务（幂等 `runDailyBackupIfDue` + 进程内 timer 挂钟 + 跳过未改动 + 启动补判 + 测试）。
4. 配置与端点（`backup-config` 读写 + `run-daily` 触发 + admin 面板「备份设置」/「删除」+ retention 显示对齐）。
5. 客户端「立即备份」按钮。
6. evergreen 文档同步。
7. 部署 + 生产验证存量裁剪。

## 13. 去风险结论（2026-06-30，plan-derisk）

**锁定方向**
- 创建：普通 push 不拍；seq 冲突 / **重叠删除（apply 前探测）** / **force-push（修漏标 protected bug）** / 手动按钮 → 拍 + protected；每日定时拍 `auto_daily`（不 protected）。
- 保留：扁平 N 天窗口（默认 7，可配）+ protected 永久 + admin 手删；删掉 GFS 周/年尾巴。
- 状态存储：备份配置 + `lastDailySeq` 存**服务端 `manifest.json` 顶层 `meta`**（服务器自有，不进同步、不进 DB 快照、不被任意设备改、启动即读）。
- 定时解耦：幂等 `runDailyBackupIfDue()` + 进程内 timer + admin 手动触发端点（外部 cron drop-in 替代）。
- 配置 UI：admin 面板（master token）；「立即备份」按钮在主 app 数据页（同步 token）。
- 范围：**一份 plan 写全**，不分阶段（内部保留侧先行）。
- 存量裁剪：**直接真删，无 dry-run 闸**；正确性靠强单测兜底。
- 孤儿：按文件名时间一并删，但**已知 protected 类前缀的孤儿不删**（防误删丢登记的受保护备份）。

**相对初版 spec 的修订/推翻**
- §5/§6 原本"小状态文件 vs manifest 元数据 / `backup-config.json` vs config 表"两可 → **锁定 manifest `meta`**。
- §5 原"挂钟 timer" → **升级为幂等函数 + timer + admin 端点**双腿解耦。
- §4 原"孤儿按时间套策略删" → **加 protected 类前缀保护闸**。
- §8 明确**无 dry-run**（曾考虑加，被用户否决）→ 转为强单测要求。
- §12 原"供后续 plan 拆分"（隐含可能分阶段）→ **单份 plan**。

**未变的既有决策**：恢复语义、ADR 0002、备份格式、不做异地备份、不做备份加密（独立轨道）。
